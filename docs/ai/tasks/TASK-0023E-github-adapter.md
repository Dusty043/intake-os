## Goal

Wire `GithubProvisioningExecutor implements ProvisioningExecutor` into the existing
`ProvisioningRegistry`. Creates a GitHub repository under the configured org with a
generated name, README, labels, and initial issues from the distribution package
subtasks.

This is an edge executor — same pattern as the Monday adapter. Does not touch the
workflow, state machine, or approval gates.

## Repository Naming

The existing `RepoNamingService` in `src/domain/repository-naming.ts` already handles
slug generation and collision detection. The GitHub executor calls this service.

Name format: `{teamPrefix}-{projectTypeSegment}-{projectSlug}`

Examples:
- `ds-dashboard-client-throughput`
- `ops-api-service-invoice-sync`
- `ds-tool-internal-crm-helper`

Rules (already implemented):
- Lowercase, hyphens only
- Max 100 characters total
- Collision detection: check GitHub API before creating; append `-2`, `-3` etc if taken
- Override: `existingRepositoryNames` passed in the provisioning plan blocks those names

## Idempotency — Same Two-Key Pattern as Monday

### 1. Internal DB idempotency key

Per-run, from TASK-0023C:
```
initial: {intakeId}:{planId}:github_repo
retry:   {intakeId}:{planId}:github_repo:retry:{retryRunId}
```

### 2. GitHub provider-level idempotency

GitHub's REST API does not have a native `Idempotency-Key` header like Monday.
Instead, idempotency is handled by **pre-checking for existence** before creating:

1. Call `GET /repos/{org}/{repoName}` before `POST /orgs/{org}/repos`
2. If the repo exists and was created by a previous provisioning run for this intake
   (verified by checking description or topics), treat it as already provisioned —
   return the existing `externalId` and `externalUrl`, mark `status: "succeeded"`
3. If the repo exists but belongs to something unrelated, treat as a collision and
   apply the collision suffix rule (`-2`, `-3`, etc.)

Stored in `ProvisioningTargetResult`:
- `externalId` = repo full name, e.g. `Simple-biz/ds-dashboard-client-throughput`
- `externalUrl` = repo HTML URL, e.g. `https://github.com/Simple-biz/ds-dashboard-client-throughput`

## GitHub API Client

Use `@octokit/rest` — already the standard GitHub REST client for Node.js.

```typescript
import { Octokit } from "@octokit/rest";
const octokit = new Octokit({ auth: process.env.GITHUB_PAT });
```

Required operations:
- `octokit.rest.repos.createInOrg` — create the repo
- `octokit.rest.repos.get` — pre-creation existence check
- `octokit.rest.repos.createOrUpdateFileContents` — write README.md
- `octokit.rest.issues.createLabel` — create standard labels
- `octokit.rest.issues.create` — create initial issues from subtasks (optional in v1)

Controlled by `GITHUB_CREATE_ISSUES=true` env var. Default: `false` for v1.

## Token Scopes Required

**Personal Access Token (classic):**
- `repo` — create private repos, push content, create issues, create labels
- If creating under an org: the token owner must be an org member with repo creation rights

**Fine-grained PAT (preferred for production):**
- Repository permissions: `Contents: Read and write`, `Issues: Read and write`, `Metadata: Read`
- Organization permissions: `Members: Read` (to verify org membership)

Validate scopes at startup using `GET /user` and `GET /orgs/{org}/members/{username}`.

---

## Error Handling

| Condition | retryable |
|---|---|
| `401 Unauthorized` | false |
| `403 Forbidden` — token lacks scope | false |
| `403 Forbidden` — org write blocked | false |
| `404 org not found` | false |
| `422 repo name taken` (unexpected collision) | false (trigger collision suffix) |
| `429 rate limit exceeded` | true |
| `503 Service Unavailable` | true |
| Network timeout | true |

For `422` on repo creation: apply collision suffix and retry with new name — this is
a domain-level retry, not a transport retry. Log the collision.

2. `GET /user` succeeds — token is valid.
3. `GET /orgs/{GITHUB_ORG}` returns the org — org name is correct.
4. Token owner is a member of the org with repo creation rights.
5. If `PROVISIONING_VALIDATE_GITHUB=true`: attempt to create and immediately delete a

## Acceptance Criteria

1. `GithubProvisioningExecutor implements ProvisioningExecutor` registered with `targetKind: "github_repo"`.
2. Runtime config validates `GITHUB_PAT` and `GITHUB_ORG` at startup before registering.
3. Uses `@octokit/rest` — not raw `fetch`.
4. Pre-checks for repo existence before creating (idempotency via check-then-create).
5. If repo already exists and matches this intake, returns it as succeeded without re-creating.
6. Applies collision suffix logic if name is taken by an unrelated repo.
7. Stores `externalId` as `{org}/{repoName}` and `externalUrl` as full GitHub URL.
8. Writes `README.md` from the distribution package on repo creation.
9. Creates standard labels on the new repo.
10. Returns `retryable: false` for auth/permission errors; `retryable: true` for rate-limit/transient errors.
11. `GITHUB_CREATE_ISSUES=true` enables creating one issue per subtask (default: false).
12. `PROVISIONING_VALIDATE_GITHUB=true` triggers smoke validation at startup.

## Relationship to Monday Adapter

Monday and GitHub are independent executors. Both register in the same
`ProvisioningRegistry`. When `PROVISIONING_TARGETS=monday,github`, both run in
parallel via `Promise.all` in `executeDistribution`. Either can fail independently;
the run status becomes `partial_success` if one fails, `completed` only if both succeed.

GitHub does not depend on Monday completing first. Order of results in
`ProvisioningRun.targets` is the order executors were registered.
